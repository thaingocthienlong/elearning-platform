<objective>
Harden support ticket submission so identity, diagnostics, and abuse limits are server-owned and suitable for serverless staging.
</objective>

<must_haves>
- Authenticated ticket submissions derive `email` and `userId` from session.
- Request-body email mismatches are rejected or ignored according to the accepted context, with no spoofed storage.
- Description, console logs, browser info, and page URL are bounded before persistence.
- Diagnostic payloads are recursively redacted for token/key/password/cookie/authorization/email-like sensitive fields.
- Ticket-specific rate limiting uses Upstash Redis when configured and a local fallback only for development/no-Redis local setup.
- reCAPTCHA test-key fallback does not mask authenticated identity protections.
</must_haves>

## Task 3: Harden support ticket route

**Type:** implementation
**Files:**
- `src/app/api/support/ticket/route.ts`
- `src/lib/request-security.ts`
- `__tests__/api/support-ticket.test.ts`

**Action:**
Update POST and GET behavior:
- require authenticated session for POST.
- derive `email` and `userId` from session.
- reject body email mismatch if a body email is provided and differs.
- check whitelist against session email.
- apply Redis/local rate limit before creating ticket.
- sanitize and bound diagnostics before persistence.
- send notification using sanitized/stored identity and description.

GET rate-limit check should avoid trusting arbitrary email for authenticated users; use session identity where possible or return unauthorized for unauthenticated checks.

**Verify:**
- `npm test -- support-ticket --runInBand`
- `npm run typecheck`

**Acceptance Criteria:**
- Spoofed support identity cannot be persisted.
- Oversized/sensitive diagnostics cannot be stored raw.
- Serverless-compatible rate limiting exists when Redis is configured.
